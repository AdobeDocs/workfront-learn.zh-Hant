# 最佳實務 — 請求佇列

## 什麼是Adobe Workfront「最佳實務」？

最佳實務准則是代表有效率、有效率的行動方針；您和您公司的使用者可輕鬆採用；並且可在您的組織內成功複製。

在檢閱這些建議時，請記住，部分Workfront最佳實務是通用的，而其他實務則可能更特定於主題。 使用這些最佳實務作為框架，以協助指導您的Workfront系統設定和使用。

## 瀏覽此頁面

捲動瀏覽本頁面時，首先您會找到該主題所有最佳實務的高層級清單。 這可讓您檢閱建議，而不需深入瞭解「原因」的詳細資訊。

「為什麼這些是最佳實務？」 區域位於高階清單之後，可讓您更詳細地瞭解部分最佳實務，以及為何將其視為流程、工具等，因此您應考慮使用Workfront執行個體進行實作。

</br>

## 請求佇列最佳實務

* 包括請求佇列每個元素的說明，這些元素包括請求佇列專案、主題群組、佇列主題和路由規則。

* 建立名為「請求佇列」或「作業」的專案狀態（等同於「目前」），以區分請求佇列專案與其他專案。

* 如果您計畫將問題核准與透過佇列提交的請求搭配使用，請建立稱為「已拒絕」的問題狀態。

* 將「通用」自訂表單指派給請求佇列，以擷取儘可能多的企業範圍一致資料。

* 避免與「每個人」共用請求佇列。 設定佇列詳細資料設定，讓使用者只看到與其需求相關的佇列。

* 建立並指派包含請求佇列報表的控制面板，讓流量管理員、系統管理員或指派的使用者可直接處理問題。

* 使用版面配置範本，針對不需要建立佇列的使用者，從專案的左側面板選單中移除請求佇列設定選項。

* 建立系統管理員請求佇列，讓使用者詢問Workfront相關問題、提出有關系統設定的請求、排程新使用者培訓等。

* 稽核請求佇列會定期識別，並取消共用未使用的佇列。

* 使用主題群組在請求佇列中組織超過10個佇列主題，以建立更短、更易於管理的清單。

* 使用主題群組和佇列主題來劃分請求佇列，而非建立多個佇列，藉此控制使用者可用的請求佇列總數。

* 為每個佇列主題設定路由規則。 至少要設定預設路由規則。

* 當需要選擇路由時，請善用主題群組和佇列主題。

* 將請求路由至團隊而非個人。

## 為什麼有這些最佳實務？


**最佳實務**

包括請求佇列每個元素的說明，這些元素包括請求佇列專案、主題群組、佇列主題和路由規則。

**原因如下**

說明可讓群組管理員、未來的系統管理員或維護請求佇列的其他人員確切瞭解請求佇列的每個部分的作用。

當您將滑鼠游標停留在新請求視窗中欄位上的資訊圖示上時，也會顯示說明資訊。

說明不一定很長，只需針對元素的用途或用途提供簡短評論。

</br>
</br>

**最佳實務**

建立名為「請求佇列」或「作業」的專案狀態（等同於「目前」），以區分請求佇列專案與其他專案。

**原因如下**

請求佇列位於專案中，且必須處於等於目前的狀態，佇列才能作用中。

為了將請求與具有「目前」狀態的實際工作專案區分開來，請建立一個狀態，以便僅用於名為「請求佇列」或「作業」的請求佇列。 然後，您可以使用此狀態來協助在撰寫報告時排除或包含請求佇列專案。

</br>
</br>

**最佳實務**

使用問題核準時建立名為「已拒絕」的問題狀態，並將「若被拒絕」選項設定為「已拒絕」狀態。

**原因如下**

藉由使用「已拒絕」狀態，可清楚表示已檢閱及拒絕要求。

</br>
</br>

**最佳實務**

將「通用」自訂表單指派給請求佇列，以擷取儘可能多的企業範圍一致資料。

**原因如下**

「通用」自訂表單會收集請求所需的標準資訊，無論提交的請求型別為何。

擁有「通用」自訂表單有助於減少您需要建立和維護的自訂表單數量。 它也能確保所有請求都以相同方式收集相同資訊，進而實現一致的報表和資料分析。

</br>
</br>

**最佳實務**

避免與「每個人」共用請求佇列。  設定佇列詳細資料設定，讓使用者只看到與其需求相關的佇列。

**原因如下**

在大多數情況下，請求佇列只需要與特定一組人員共用，例如團隊、廠商、客戶等。 當請求者在請求佇列中只看到他們需要的內容時，可讓您輕鬆找到並導覽。

</br>
</br>


**最佳實務**

建立並指派包含請求佇列報表的控制面板，讓流量管理員、系統管理員或指派的使用者可直接處理問題。

**原因如下**

讓使用者快速輕鬆地存取傳入的請求，意味著工作不會在隨機播放中遺失。

</br>
</br>

**最佳實務**

使用版面配置範本，針對不需要建立佇列的使用者，從專案的左側面板選單中移除請求佇列設定選項。


**原因如下**

這可確保所有請求佇列都經過適當的建立程式（例如由治理委員會稽核），並由系統或群組管理員正確設定。

此外，這有助於讓佇列清單保持井然有序，並專注於您組織需求的請求型別。

</br>
</br>

**最佳實務**

建立系統管理員請求佇列，讓使用者詢問Workfront相關問題、提出有關系統設定的請求、排程新使用者培訓等。

**原因如下**

這可讓使用者集中提交問題，而管理員則可收集、監控及回應Workfront相關問題。

此外，此資訊可用於顯示系統管理員角色的時間、努力和價值，以及作為額外系統管理員的理由。

</br>
</br>


**最佳實務**

稽核請求佇列會定期識別，並取消共用未使用的佇列。

**原因如下**

定期稽核Adobe Workfront系統中的設定和專案，有助於保持系統整齊，並且沒有不需要的專案。 如果佇列不再被使用或監視，請確定使用者無法再存取它，以免工作請求變成無效。

</br>
</br>


**最佳實務**

使用主題群組在請求佇列中組織超過10個佇列主題，以建立更短、更易於管理的清單。

**原因如下**

主題群組可減少要選取的選項初始清單，以增加使用者採用率，並減少混淆。 這可讓使用者在嘗試提交請求時，輕鬆找到他們想要的東西，而不會讓他們不知所措。

此外，它讓系統管理員和/或請求佇列管理員有機會為使用者建立順暢的導覽路徑，以及更有效地組織和報告提交的請求型別。

</br>
</br>

**最佳實務**

使用主題群組和佇列主題來劃分請求佇列，而非建立多個佇列，藉此控制使用者可用的請求佇列總數。

**原因如下**

太多請求佇列使得使用者難以找到他們需要的內容。

更少的佇列還有助於流量協調員、系統管理員或其他管理佇列的人，讓他們更快找到所需的資訊，而無需導覽至多個請求佇列專案。

如果不同的請求佇列需要不同的存取權，或是如果合併佇列會讓使用者感到困惑，請建立多個請求佇列。

</br>
</br>

**最佳實務**

為每個佇列主題設定路由規則。 至少要設定預設路由規則。

**原因如下**

路由規則可確保某人一律會獲得傳入請求的指派，因此工作不會受阻。

</br>
</br>

**最佳實務**

當需要選擇路由時，請善用主題群組和佇列主題。

**原因如下**

路由規則無法套用至自訂表單的欄位。 因此，如果需要將不同型別的請求路由到不同的團隊/個人，則請使每種型別的請求都有自己的主題群組/佇列主題，以便工作可以正確路由。

</br>
</br>

**最佳實務**

將請求路由至團隊而非個人。

**原因如下**

當請求被傳送給團隊時，它使整個團隊能夠檢視正在提出的請求以及即將進行的工作可能包含的內容。 每個人都可以觀看「團隊」頁面上的新專案，或追蹤控制面板上報告的新增功能。

它也能確保當流量管理員或其他負責檢閱或指派傳入請求的人無法取得時，備份會自動可用，並可存取請求資訊。
