# 最佳做法 — 組織單位

## 什麼是Adobe Workfront的「最佳實務」？

最佳做法是代表有效、有效行動方針的准則；易於被您和公司用戶採納；並可在貴組織中成功複製。

在您檢閱這些建議時，請記住，有些Workfront最佳實務是通用的，有些則可能是更具體的主題。 將這些最佳實務作為架構，協助您指導Workfront系統的設定與使用。

## 瀏覽此頁面

當您捲動此頁面時，首先會找到主題所有最佳實務的高階清單。 這可讓您檢閱建議，而不需深入了解「原因」的詳細資訊。

「這些最佳實務為何？」 在高階清單後找到的「 」區域，會更詳細說明某些最佳實務，以及為何這些作法被視為程式、工具等，您應考慮使用Workfront執行個體實作。

</br>
</br>

## 組織單位最佳做法

* 在Workfront中設定組織單位之前，請先識別關鍵報表類別和需求。
* 定期評估您現有的Workfront組織結構，確保其符合您的需求。
* 每季審核您的用戶群，或至少每6個月審核一次。

* 識別並訓練使用者成為群組管理員。

* 與使用者溝通Workfront的運作狀況、Workfront功能將有哪些變更，或流程/工作流程更新。

* 內部使用者通常只需要一間公司。

* 為您要在Workfront執行個體中追蹤或需要執行個體登入權限的外部用戶端或廠商建立公司。

* 將Workfront群組保持至最少並與組織的部門建立關聯。

* 使用組來表示部門結構，使用組來表示工作結構。
* 運用群組管理員協助系統管理員管理使用者和設定。

* 允許組管理員建立自己的子組。

* 將版面範本指派給主團隊。

* 使用者的主團隊應是他們最常合作的團隊。

* 使用較大的組織單位來與人員集共用項目。

</br>
</br>


## 這些最佳實務為何？

**最佳實務**

在Workfront中設定組織單位之前，請先識別關鍵報表類別和需求。



**原因如下**

了解您要尋找的量度，以及您想在Workfront中收集哪些使用者及其所執行工作的相關資訊，讓您能夠專注於建立符合這些需求的組織單位。

</br>
</br>

**最佳實務**

定期評估您現有的Workfront組織結構，確保其符合您的需求。



**原因如下**

隨著您的組織成長，或隨著Workfront在您的組織內部擴充，您將會找到新的不同方式，以提升流程效率以及使用者與工作的關係。 因此，定期評估組織單位的結構，可以讓您知道該結構是否仍然有效，或者是否需要更新。

</br>
</br>


**最佳實務**

每季審核您的用戶群，或至少每6個月審核一次。



**原因如下**

隨著您的組織逐漸使用Workfront並增加工具的使用，資訊會在系統中建置。 這包括使用者人數、使用的授權數量、已建立多少公司、群組或團隊等。



隨著時間推移，您的組織也可能會經歷變更。 比如，上個季度可能發生了部門重組。 或者一些員工決定看到其他機會。 這表示Workfront中的使用者正在取用授權，而且團隊或群組中可能會出現差異，而這些差異可能會影響資源估算或工作指派。



對Workfront中的組織單位和使用者進行每季的稽核，是維持系統順暢有效運作的關鍵。 這也是確保Workfront利用率穩定或增長的好時機，而且員工正在使用Workfront進行持續的工作管理，而不是電子郵件或電子錶格。

</br>
</br>

**最佳實務**

識別並訓練使用者成為群組管理員。



**原因如下**

選擇對Adobe Workfront及其群組使用系統的方式有深入了解的人員，以及有時間專門擔任群組管理員的人員。



組管理員可以設定狀態、修改項目首選項，或為他們管理的組和子組建立時間表配置檔案。 他們也可以管理其群組的使用者、調整存取層級並指派版面範本。



這樣，系統管理員就不必再進行某些日常設定和維護Workfront。 這也能讓每個群組更能控制自己的設定和作業偏好設定，而不會干擾Workfront中的其他群組。

</br>
</br>


**最佳實務**

內部使用者通常只需要一間公司。



**原因如下**

Workfront公司應適用於整個組織，而不只是業務領域。 如果您將內部使用者分割為不同的公司，可能會造成在Workfront中運作的權限問題。 Workfront群組是組織內部使用者的慣用方法。

</br>
</br>

**最佳實務**

為您要在Workfront執行個體中追蹤或需要執行個體登入權限的外部用戶端或廠商建立公司。



**原因如下**

與外部廠商建立公司可讓您允許讓這些廠商進入Workfront，以檢閱為他們建立的內容或產品的進度，同時限制其在系統中的存取權。



Workfront存取權是透過指派給供應商公司使用者的物件共用和存取層級組合而授予。 例如，您可以設定限制，讓這些使用者只能看見自己公司中的人員，和/或授與特定項目（例如請求佇列和請求）的權限，讓特定公司的使用者才能看見他們。 這有助於建立您在Workfront中保留的資訊的隱私權。

</br>
</br>


**最佳實務**

將Workfront群組保持至最少並與組織的部門建立關聯。



**原因如下**

在Workfront管理工作的部門大多需要小組。 例如，行銷部門可能需要自己的群組，但來自其他部門(只在Workfront中請求工作的)的使用者，可以放在預設群組中。

</br>
</br>

**最佳實務**

使用組來表示部門結構，使用組來表示工作結構。



**原因如下**

首頁群組是使用者和專案的必填欄位。 這為組織和報告用戶及其參與的高級工作創造了良好的方式。



但是，當要對一組人員進行分配時（通過項目上的任務或通過請求隊列），只能對團隊進行分配。

</br>
</br>

**最佳實務**

運用群組管理員協助系統管理員管理使用者和設定。



**原因如下**

系統管理員可存取Workfront中的所有項目，包括全域系統設定。 設定組管理員可以訪問的設定由系統管理員控制，並且僅應用於該特定組。



由群組管理員可讓系統管理員委派許多責任，讓他們專注於大局項目，而非Workfront的日常維護工作。 群組管理員可更輕鬆地聯絡其群組的需求，為使用者提供更好的服務。

</br>
</br>


**最佳實務**

允許組管理員建立自己的子組。



**原因如下**

組管理員可以建立和管理自己的子組。 雖然系統管理員可以建立子組，但將此分配給組管理員可以節省大量時間。 此外，群組管理員將更熟悉群組和子群組的需求和工作流程，因此也能為群組成員提供更理想的Workfront體驗。

</br>
</br>

**最佳實務**

將版面範本指派給主團隊。



**原因如下**

請務必確定您的團隊在運作時處於相同頁面。 這表示他們可更輕鬆存取相同的報表和工具。 版面範本可讓您建立該環境，並增強（甚至可能改善）您的團隊已有的流程。

</br>
</br>

**最佳實務**

使用較大的組織單位來與人員集共用項目。



**原因如下**

在Workfront中共用項目時，使用較大的組織單位，例如群組和團隊 — 專案、報表等。 — 表示您可以快速輕鬆地將對象與選定的一組人員同時共用。 將個別使用者的名稱新增至共用視窗可能既麻煩又費時。



這也可讓您更輕鬆地維護項目上的共用。 隨著團隊成員在Workfront中更新，項目的共用權限會自動更新，授予新團隊成員對項目的訪問權並移除前團隊成員的訪問權。
